## 1. Schema summary

The prejoined modeling table contains one row per (individual_id, index_dt) observation. It merges:
- Tabular clinical/claims features (all available before index_dt, post-leakage-exclusion)
- New TE embedding features (columns starting with `embedding_`; exact join on individual_id + index_dt)
- Binary outcome ip6 (1 = ≥6 inpatient days in 6 months post index_dt)
- Split key ind_id_last_digit (deterministic digit of hashed individual_id)

## 2. Availability table (narrative)

All features are synchronous with index_dt — available at prediction time by construction. The embedding columns are generated from the same formal-training cohort using the same anchor date; the matched_embedding_index_dt column records the join audit timestamp and is excluded from features. OOT rows (index_dt > 2025-06-30) are in the table but excluded from in-time splits; prepare.py filters them out of train/val/test.

## 3. Leakage audit summary

Leakage audit performed 2026-04-24. Excluded columns: all cost/claim amount columns (which include post-index spend), outreach/contact flags (post-index intervention indicators), eligibility flags used for cohort inclusion logic, and the audit column matched_embedding_index_dt. The EXCLUDE_COLUMNS list in prepare.py matches the notebook verbatim. Any additions require C3.

## 4. Transformations applied pre-agent

- V1–V28 style PCA: not applicable (this is claims/clinical data, not credit-card PCA)
- New TE embeddings are generated by a separate model (not part of this campaign's scope)
- The modeling table is produced by a SQL pipeline that handles the join and leakage exclusion upstream

## 5. Known data quality issues

- Some members may appear multiple times at different index_dt values (multiple observation dates)
- Deduplication is applied in prepare.py: keep last row per (individual_id, index_dt) to handle any upstream duplicates
- Negative class is highly prevalent (~95%+); training set uses 10:1 negative downsampling to match baseline pipeline
