# Vertex Model Trainer

An end-to-end Vertex AI pipeline that trains an XGBoost binary classification model using a **prebuilt Vertex AI container image**, registers the trained model to the **Vertex AI Model Registry**, and immediately runs **batch predictions** on the holdout dataset — all without requiring a custom Docker build.

---

## Overview

| Attribute | Value |
|---|---|
| **Model type** | XGBoost binary classification (`binary:logistic`) |
| **Training image** | `us-docker.pkg.dev/vertex-ai/training/xgboost-cpu.2-1:latest` |
| **Serving image** | `us-docker.pkg.dev/vertex-ai/prediction/xgboost-cpu.2-1:latest` |
| **GCP region** | `us-east4` |
| **Data source** | BigQuery (`anbc-hcb-dev.clin_analytics_hcb_dev`) |
| **Output** | Vertex AI Model Registry entry + BigQuery predictions table |
| **Package version** | `0.1.0` |

### Why a prebuilt image?

Using the same XGBoost version (`2.1.4`) in both the training image and the serving image guarantees that the saved `model.bst` artifact is fully compatible with Vertex AI Batch Prediction. The model is intentionally saved **without feature names** (from numpy arrays) so the serving container can accept plain numeric arrays as input — a strict requirement of the Vertex AI Batch Prediction API.

## Training Pipeline — Step by Step

`trainer/task.py` orchestrates 17 sequential steps when the Vertex AI Custom Training Job runs:

| Step | Description |
|---|---|
| 1 | Parse CLI arguments — all complex configs passed as JSON or base64-encoded JSON |
| 2 | Set up GCP environment variables and random seeds |
| 3 | Initialise BigQuery client |
| 4 | Load **outcome**, **training features**, and **holdout features** from BigQuery |
| 5 | Fill missing values (embeddings & numerics → `0`, strings → `''`) |
| 6 | Apply categorical encodings (`gender_cd`, `drug_ind` binary; `region` one-hot) |
| 7 | Inner-join outcome onto features; set `individual_id` as the DataFrame index |
| 8 | Stratified train / test / val split (80 / 10 / 10 by default) |
| 9 | Load pre-selected feature list from inline JSON or GCS file |
| 10 | Build `xgb.DMatrix` objects from **numpy arrays** — no feature names stored |
| 11 | Train with `xgb.train()` (3,101 rounds, Optuna-tuned hyperparameters) |
| 12 | Evaluate: ROC-AUC + Lift / PPV / Sensitivity at top 1% and top 10% |
| 13 | Save `model.bst` locally and upload to `gs://<bucket>/<prefix>/model/model.bst` |
| 14 | Stream `model_metadata.json` directly to GCS (no temp file) |
| 15 | Upload holdout predictions to BigQuery (optional) |
| 16 | Register model (or new version) in Vertex AI Model Registry |
| 17 | Trigger Vertex AI Batch Prediction job on the test split |

## Quick Start

### 1. Configure `config.yaml`

All pipeline settings are controlled from `config.yaml`. At minimum, review and update:

```yaml
gcp:
  project: "anbc-dev-hcm-cm-de"       # Vertex AI billing project
  gcp_project: "anbc-hcb-dev"         # BigQuery project
  gcp_db: "clin_analytics_hcb_dev"    # BigQuery dataset
  bucket_name: "hcm-cm-de-code-hcb-dev"

vertex_ai:
  service_account: "gchcb-hcm-cm-de-ontpd@anbc-dev-hcm-cm-de.iam.gserviceaccount.com"
  cmek_key: "projects/cvs-key-vault-nonprod/..."

model_registry:
  display_name: "ss-xgboost-model"    # leave blank to skip registration

batch_prediction:
  enable: true
  output_table: "anbc-hcb-dev.clin_analytics_hcb_dev.model_test_predictions"
  compute_dataset: "hcm_cm_de_beam_dev_hcm_cm_de"
```

### 2. Run the Orchestration Notebook

Open `model_training.ipynb` and run all cells. The notebook will:

1. Load `config.yaml` into Python dicts
2. Build and upload the `ss-models-trainer` Python package to GCS
3. Assemble the full CLI argument list (including base64-encoded SQL and hyperparameters)
4. Compile the KFP pipeline to `model_training_pipeline.json`
5. Submit the pipeline to Vertex AI and wait for completion

## Configuration Reference

### `config.yaml` Top-Level Sections

| Section | Purpose |
|---|---|
| `gcp` | GCP project IDs, BigQuery dataset, GCS bucket |
| `vertex_ai` | Region, service account, CMEK key, Docker image, machine type |
| `bigquery_labels` | Labels applied to BQ tables created by the pipeline |
| `sql_queries` | Three SQL queries: `outcome_query`, `train_features_query`, `holdout_features_query` |
| `data_processing` | Random seed, outcome variable, index column, split ratios |
| `categorical` | Binary mappings (`gender_cd`, `drug_ind`) and one-hot encoding (`region`) |
| `selected_features` | Path to a pre-selected feature list file (local or GCS) |
| `model` | XGBoost hyperparameters and number of boosting rounds |
| `metrics` | Percentile thresholds for lift/PPV/sensitivity evaluation |
| `output` | Optional BigQuery table for holdout predictions |
| `model_registry` | Model Registry display name, serving container, versioning settings |
| `batch_prediction` | Output table, compute dataset, machine type, expiration |
| `model_labels` | Identity and governance labels attached to the Registry entry |
| `environment_variables` | Environment variables injected into the Vertex AI training container |

### SQL Query Structure

The pipeline requires three SQL queries that must return:

| Query key | Required columns |
|---|---|
| `outcome_query` | `individual_id`, `<outcome_var>` |
| `train_features_query` | `individual_id` + all feature columns (`WHERE train_test_split <= 75`) |
| `holdout_features_query` | `individual_id` + all feature columns (`WHERE train_test_split > 75`) |

Outcome columns (e.g. `Oncologic_case_post`) must be excluded from feature queries.

## Model Compatibility with Prebuilt Images

The key design constraint that ensures batch prediction works:

1. **Training image** (`xgboost-cpu.2-1`) and **serving image** (`xgboost-cpu.2-1`) use the **same XGBoost version** (`2.1.4`).
2. `xgb.DMatrix` is built from **numpy float32 arrays** — not DataFrames — so no feature names are embedded in the saved booster.
3. The saved `model.bst` sits alone in its GCS directory (`gs://<bucket>/<prefix>/model/`) so the Model Registry artifact URI resolves to exactly one file.
4. The `model_metadata.json` stored alongside `model.bst` records `feature_names_in_order` — the exact column order required by callers of the serving endpoint.

## Model Registration

After training, `trainer/model_card.py` registers the model in the Vertex AI Model Registry with:

- **Artifact URI**: `gs://<bucket>/<prefix>/model/` (contains `model.bst`)
- **Serving container**: `us-docker.pkg.dev/vertex-ai/prediction/xgboost-cpu.2-1:latest`
- **Labels**: up to 64 Vertex AI labels covering identity, governance, hyperparameters, and evaluation metrics
- **Serving container env vars**: all fields from `model_metadata.json` (including `FEATURE_NAMES` as a JSON array)

To upload a **new version** of an existing model instead of creating a new one, set:

```yaml
model_registry:
  upload_to_existing_model: true
  existing_model_resource_name: "projects/my-project/locations/us-east4/models/1234567890"
```

## Batch Prediction

`trainer/batch_predict.py` runs a BigQuery-to-BigQuery batch prediction job immediately after model registration:

**Flow:**
1. Build a test DataFrame from `X_test` (features) + `y_test` (actual labels)
2. Upload to a temp input table in the compute project
3. Submit a Vertex AI Batch Prediction job via REST API — `excludedFields` prevents the ID and actual-label columns from being sent to the model
4. Wait for the job to complete
5. Copy results to the final output table
6. Delete temp tables

**Output table schema:**

| Column | Description |
|---|---|
| `individual_id` | Member identifier |
| `actual_<outcome_var>` | Ground truth label from the test split |
| `<feature_col_1>` … | Selected feature values |
| `prediction` | Model output probability (positive class) |

## Key Files Reference

### `pipeline_helpers.py`

Helper functions used by the orchestration notebook. Not normally modified by users.

| Function | Purpose |
|---|---|
| `build_package()` | Runs `python setup.py sdist` to create the `.tar.gz` package |
| `upload_file_to_gcs()` | Uploads a local file to a `gs://` URI |
| `build_and_upload_package()` | Combines build + upload in one call |
| `create_model_training_job_component()` | Creates a KFP `CustomTrainingJobOp` component |
| `_encode_json_args()` | Base64-encodes JSON CLI args to avoid payload escaping issues |

### `trainer/task.py`

The main entry point invoked by the Vertex AI container. All settings are passed via CLI flags; no hardcoded values. Run `python -m trainer.task --help` for the full argument list.

### `trainer/model_card.py`

Owns everything after training: label assembly, `model.bst` save, `model_metadata.json` upload, and Model Registry registration. The `feature_names_in_order` list in metadata is the source of truth for column ordering.

### `trainer/batch_predict.py`

Orchestrates the Vertex AI Batch Prediction job. Uses `X_test` (the same split used for evaluation) as input so predictions can be compared against known labels.

### `utils/helpers.py`

Shared utilities used across multiple trainer modules. Includes `sanitize_vertex_label()` (enforces Vertex AI label constraints) and `build_model_labels()` (assembles the full ≤64-label dict).

## Governance & Labels

All Vertex AI Model Registry entries are tagged with the following label categories (up to Vertex AI's 64-label limit):

- **Identity**: `model_name`, `tenant`, `self_serve`, `vertex_model_id`
- **Governance**: `owner`, `lob`, `costcenter`, `unique_id`, `pipeline_type`
- **Data provenance**: `outcome_var`, `bq_project`, `bq_dataset`
- **Dataset sizes**: `train_rows`, `test_rows`, `val_rows`, `num_features`
- **Hyperparameters**: `num_boost_round`, `max_depth`, `learning_rate`, `subsample`, `colsample_bytree`, `tree_method`
- **Evaluation**: `roc_auc`, `lift_1_perc`, `ppv_1_perc`, `sensitivity_1_perc`, `lift_10_perc`, `ppv_10_perc`, `sensitivity_10_perc`
- **Provenance**: `training_date` (YYYY-MM-DD)
